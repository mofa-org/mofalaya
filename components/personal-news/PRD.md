# PRD｜个人新闻联播（Personal News Broadcast Script Generator）

## 1. 产品背景与目标

### 背景

信息来源高度碎片化（RSS 新闻、X、YouTube、Gmail、日程、天气），用户每天需要在多个 App 间切换，**获取信息成本高，且缺乏整体叙事**。

传统「信息聚合」停留在列表层面，**无法形成可直接“听”的内容**。

### 产品目标

构建一个 **个人化“新闻联播”内容生成 Agent**：

- 自动聚合与用户高度相关的信源
- 将碎片化信息 **重写为结构化、可播报的新闻稿**
- 输出 **纯文本新闻播报稿**，供下游 TTS / 播报 Agent 使用
- 强调 **秩序感、权威感、节奏感**（而非聊天摘要）

### 非目标（明确不做）

- ❌ 不做 TTS / 音频合成
- ❌ 不做播放端 / App
- ❌ 不做社交发布
- ❌ 不做 LinkedIn 数据接入

------

## 2. 目标用户

- 信息密集型用户（创始人 / PM / 研究人员 / 投资人 / 内容创作者）
- 有固定「早间 / 晚间信息摄入习惯」
- 希望 **“被播报”而不是“自己刷”**
- 已或计划使用多 Agent 工作流（如你当前的 MoFA / Agent 架构）

------

## 3. 核心使用场景

### 场景 1｜早间新闻联播（默认）

> “8:30，生成一条 3 分钟的个人新闻联播稿”

内容包含：

- 今日重点新闻（RSS）
- 与我相关的 X 动态
- 今日关键日程
- 今日天气
- 邮件中需要我关注的事项

### 场景 2｜通勤快报（精简版）

> “只要最重要的 3 条 + 天气 + 日程”

### 场景 3｜晚间复盘（可选扩展）

> “今天发生了什么、我需要记住什么”

------

## 4. 信息源（Inputs）

### 4.1 RSS 新闻网站

**用途**：公共新闻 / 行业新闻 / 全球事件

- 用户可配置多个 RSS 源（如：
  - 综合新闻
  - 科技 / AI
  - 财经
  - 自定义博客）
- 系统按「新鲜度 + 重要性」筛选

**输入格式（示例）**

```
{
  "source": "rss",
  "items": [
    {
      "title": "...",
      "summary": "...",
      "published_at": "...",
      "source_name": "..."
    }
  ]
}
```

------

### 4.2 X（Twitter）

**用途**：你“关注的人”在说什么

- 仅限：
  - 你关注的人
  - 你被提及 / 回复
- 不做全量时间线
- 强调 **与你相关、你关心的人**

**可选筛选规则**

- 转推数 / 点赞数阈值
- 是否包含链接
- 是否来自“重点关注列表”

**输入格式（示例）**

```
{
  "source": "x",
  "items": [
    {
      "author": "username",
      "text": "...",
      "engagement": {
        "likes": 120,
        "retweets": 30
      },
      "created_at": "..."
    }
  ]
}
```

------

### 4.3 Gmail 邮箱

**用途**：现实世界对你的“直接触达”

#### 关注类型

- 未读邮件
- 来自重要联系人 / 域名
- 包含明确 action（会议、确认、截止）

#### 不做

- 营销邮件
- 大规模订阅（这些走 RSS）

**输入格式（示例）**

```
{
  "source": "gmail",
  "items": [
    {
      "from": "xxx@company.com",
      "subject": "...",
      "snippet": "...",
      "received_at": "..."
    }
  ]
}
```

------

### 4.4 Calendar（日程）

**用途**：今天你“要面对什么”

- 今日 / 未来 24h 事件
- 标记重要级别（如 title 中包含关键字）

**输入格式**

```
{
  "source": "calendar",
  "items": [
    {
      "title": "...",
      "start": "...",
      "end": "...",
      "location": "..."
    }
  ]
}
```

------

### 4.5 天气

**用途**：生活服务信息，增强“新闻联播真实感”

- 今日天气 + 极端提醒
- 可基于当前定位或默认城市

------

## 5. 核心功能：新闻播报稿生成

### 5.1 输出形式（唯一输出）

👉 **结构化、可直接播报的纯文本新闻稿**

- 无代码
- 无 Markdown
- 无 bullet point
- 适合直接喂给 TTS Agent

------

### 5.2 新闻播报结构（强约束）

#### 固定栏目顺序

1. 片头（可选，文本版）
2. 今日要闻（RSS）
3. 与我相关的动态（X / Gmail）
4. 日程速递
5. 生活服务（天气）
6. 结束语

------

### 5.3 播报风格规范（非常重要）

**整体**

- 冷静、克制、第三人称
- 类似「新闻联播 / 晨间简报」
- 不使用“你”“我看到”

**允许**

- “今天值得关注的是……”
- “需要注意的是……”
- “接下来是与你相关的动态……”

**禁止**

- 口语化闲聊
- 情绪化评价
- 网络流行语
- 过度 AI 味总结

------

### 5.4 示例输出（节选）

> 现在是 2026 年 1 月 20 日，以下是为你整理的个人新闻联播。

> 今日要闻方面，多家科技媒体关注到……（RSS 摘要转述）

> 与你相关的动态中，你关注的账号 @xxx 在 X 上表示……该内容在短时间内获得了较高讨论度。

> 邮件方面，你收到一封来自……的邮件，对方提到需要你在本周内确认相关事项。

> 日程速递显示，今天上午十点你将参加……会议，下午三点还有一场……

> 生活服务方面，你所在地区今天多云转小雨，最高气温……外出建议携带雨具。

> 以上是今天的个人新闻联播，祝你今天进展顺利。

------

## 6. 核心规则（Agent Prompt 级别）

### 6.1 信息筛选规则

- **宁少勿滥**
- 每个栏目默认 ≤ 3 条
- RSS 与 X 不允许重复表达同一事件

### 6.2 重写规则

- 不直接朗读标题
- 必须转述、压缩、新闻化
- 每条控制在 **15–30 秒播报长度**

### 6.3 隐私规则

- 邮箱地址、手机号模糊处理
- 仅保留必要身份信息

## 8. 系统边界 & Agent 协作

### 本 Agent 负责

- 聚合数据
- 筛选、排序
- **生成最终播报稿**

### 下游 Agent 负责

- TTS
- 声音风格
- 播放 / 分发
- 多角色播报（如需要）

------

## 9. 成功指标（Product Metrics）

- 用户无需再“看”信息
- 播报稿可直接使用，无需人工改写
- 单次播报时长可控（2–4 分钟）
- 信息“有秩序感”，而非堆砌

------
